# CRM 半年競賽週期設計討論

> 紀錄日期：2026-05-14
> 狀態：**尚未實作**，僅為討論結論，等需求成熟再來動工。

## 一、需求描述

業務競賽以**半年為一個週期**（H1：1/1–6/30，H2：7/1–12/31）。
進入新週期時，希望讓 CRM 有「重新開始」的感覺：

- 所有客戶的「**目前階段**」回到「尚未聯絡」
- 但「**拜訪紀錄（歷史紀錄）**」完全保留，可以往回翻看上一期談過什麼、推到哪
- 業務漏斗的計分也對應到當期，每期重新累積

換句話說：**「目前階段」是綁定競賽週期的，「拜訪紀錄」是綁定時間軸的**。

## 二、行為範例

同一位客戶在跨期時應該長這樣：

| 時間 | 動作 | account.stage（目前） | 拜訪紀錄 |
|---|---|---|---|
| 2026/3/15 | 初訪 | 初步聯繫 | +1 筆（H1） |
| 2026/5/20 | 成交 | 成交 | +1 筆（H1） |
| 2026/6/30 | H1 結算 | 成交 | （H1 共 2 筆） |
| 2026/7/1 | 進入 H2，自動重置 | **尚未聯絡** | （H1 那 2 筆仍在，可查閱） |
| 2026/8/10 | H2 再次拜訪 | 初步聯繫 | +1 筆（H2） |
| 2026/10/5 | H2 成交 | 成交 | +1 筆（H2） |

業務漏斗在 H1 看：初步聯繫 1、成交 1。
業務漏斗在 H2 看：初步聯繫 1、成交 1。
兩期分別計分，互不干擾。

## 三、與目前實作的差異

### 現況

- 客戶身上只有單一的 `account.stage`，是全域唯一的「當前階段」狀態
- 業務漏斗的「半年」只是**事件過濾**：只把日期落在區間內的拜訪紀錄、legacy 匯入抓出來統計，**完全不會改動 `account.stage`**
- CRM 拜訪頁面總是顯示客戶的「現況」階段，不分週期

### 與需求的落差

- 沒有「**這個客戶在 H1 是什麼階段、H2 是什麼階段**」的概念
- 進新週期不會自動重置
- CRM 列表沒有週期視角

## 四、未來實作的資料模型草圖

需要把「階段」從客戶身上的單一屬性，**升維成「客戶 × 競賽週期」的二維紀錄**。

### 可能的資料結構

```js
// 新增一張表（或 account 上的 map）：每個客戶在每個競賽週期的階段
accountCycleStages = [
  { accountId, cycleId: "2026H1", stage: "成交", updatedAt },
  { accountId, cycleId: "2026H2", stage: "尚未聯絡", updatedAt },
  ...
]

// 競賽週期定義
cycles = [
  { id: "2026H1", start: "2026-01-01", end: "2026-06-30" },
  { id: "2026H2", start: "2026-07-01", end: "2026-12-31" },
  ...
]
```

`account.stage` 可保留為「**當期階段**」的快取，或拿掉改成從 `accountCycleStages` 動態查當前週期。

### 衍生邏輯

- **進新週期**：每位客戶自動寫一筆 `{ cycleId: 新, stage: "尚未聯絡" }`，不動 `account.stage` 以外的任何資料
- **拜訪紀錄推進階段**：寫到目前週期的那筆 `accountCycleStages`，而不是直接改 `account.stage`
- **業務漏斗統計**：可選擇用「事件來源」（visit.stageAfter，現況）或「週期狀態」（accountCycleStages.stage）
- **CRM 列表**：預設顯示「當期階段」，可切換看歷史週期的快照

## 五、舊資料遷移

- 目前所有客戶只有 `account.stage`
- 遷移時可選擇：
  - **方案 A**：把 `account.stage` 視為「最近一期」的階段，補上對應 `cycleId` 的紀錄。先前的週期都當作沒紀錄（或全部標「尚未聯絡」）
  - **方案 B**：依拜訪紀錄日期回推，每個週期裡看最後一筆 visit 的 stageAfter，補出當期階段
- legacy `sourceRaw` 的多階段勾選 → 該怎麼分配到哪幾期？需要再想

## 六、待決定的開放問題

實作前要先把這些釐清，否則邏輯會卡：

1. **跨期顯示策略**：CRM 列表預設顯示「當期階段」，那已經跨多期的老客戶要不要在「全部」視圖看到完整歷程？
2. **重置時機**：自動於 7/1 / 1/1 凌晨重置，還是「進入新週期看頁面時」才動？前者要排程，後者比較簡單
3. **「成交」的歸屬**：一個客戶 H1 成交、H2 又成交，是兩件還是一件？（影響業績統計）
4. **保服 / 理賠 / 轉介紹**：這些「成交後階段」要不要每期也重置？還是不重置（一直跟著客戶）？
5. **遷移既有資料**：用方案 A 還是 B？
6. **業務漏斗的「最多發生」「成交數」**：用事件來算（現況）還是用週期狀態來算？
7. **拜訪紀錄與週期的關係**：visit 是不是要寫死 `cycleId`，方便往後查詢？還是依日期動態歸屬？
8. **CRM 篩選 vs 漏斗統計**：兩者要怎麼對齊？是否一律以「當期階段」為準？
